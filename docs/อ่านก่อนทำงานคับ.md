## Backend — มีทั้งหมด 9 services (รวม Gateway)

| Service | หน้าที่ | DB ที่เป็นเจ้าของ |
|---|---|---|
| `api-gateway` | จุดเดียวที่ frontend ทุกตัวเรียก, route ต่อไปยัง service ภายในผ่าน Service Binding | — (ไม่มี DB) |
| `auth-svc` | verify JWT จาก Kinde, middleware auth กลาง | — (พึ่ง Kinde) |
| `user-svc` | users, customers, staff, admins, addresses | User-svc |
| `product-svc` | products, brands, categories, productImages, bundles, **inventory + inventoryLogs** | Product-svc |
| `cart-svc` | carts, cartItems | Redis (Upstash) |
| `order-svc` | orders, orderItems, shipments | Order-svc |
| `payment-svc` | payments, เชื่อม Omise | Payment-svc |
| `notification-svc` | **QStash subscriber** — รับ event แล้วส่งอีเมลผ่าน Resend | Notification-svc |
| `report-svc` | **QStash subscriber เหมือนกัน** — รับ event ชุดเดียวกัน (fan-out ผ่าน QStash Topic) มาสะสมเป็นตัวเลขสรุป แทนที่จะ query หนักทุกครั้งที่เปิด dashboard | Report-svc (aggregate) |
| `inventory-svc` | แยก service สำหรับจัดการสต็อกโดยเฉพาะ (อาจทำทีหลังถ้ารู้สึก product-svc ใหญ่ไป) | Inventory-svc |

> `notification-svc` กับ `report-svc` ฟัง event ชุดเดียวกัน (`stock.updated`, `order.status_changed`, `payment.success`) ผ่าน QStash Topic เดียว แต่เป็น subscriber คนละตัว อิสระจากกัน — ตัวไหนเขียน endpoint เสร็จก่อนก็เริ่มรับ event ได้เลย ไม่ต้องรอกัน และ **publisher (บุญ/เดียร์) เขียนโค้ด publish ได้ตั้งแต่ Phase 2 โดยไม่ต้องรอ subscriber เสร็จก่อน** เพราะ QStash retry ให้อัตโนมัติถ้าปลายทางยังไม่มี endpoint จริง — แค่ต้องตกลง payload format ใน `packages/types` ให้ตรงกันตั้งแต่ต้น (เขตเป็นเจ้าของทั้ง `api-gateway` และ `report-svc` เพิ่ม แต่ทั้งคู่ทำทีหลังได้เพราะไม่ block คนอื่น — gateway แค่ proxy เฉยๆ ทำหลัง service จริงเริ่มมีบ้างแล้วก็ได้)

## การแบ่งคนรับผิดชอบ

| คน | Service ที่รับ | เหตุผล |
|---|---|---|
| **เขต** | `api-gateway`, `auth-svc`, `user-svc`, `report-svc` | foundational ที่ต้องเสร็จก่อนคนอื่นเริ่ม test ได้ (auth/user) + gateway/report-svc ทำทีหลังได้เพราะไม่ block ใคร (gateway แค่ routing, report-svc เป็น subscriber อิสระ) ภาระเบาเข้ากับงาน infra |
| **บุญ** | `cart-svc`, `order-svc`, `payment-svc` | สาย "ซื้อของ" ที่เรียงต่อกันเป็น pipeline เดียว (cart → order → payment) คนเดียวคุมทั้ง flow ลด friction ในการประสานงาน |
| **เดียร์** | `product-svc`, `inventory-svc` (รวม bundle), `notification-svc` | สาย "จัดการสินค้า/สต็อก" ที่ staff ใช้งานหนักสุด + notification ที่ต้องฟัง event จากทั้งฝั่ง product และฝั่ง order |

### จุดที่บุญกับเดียร์ต้องคุยกันโดยตรง (ไม่ผ่านเขต)

ตามที่ขอ — เอา service ที่ผูกกันแน่นให้ 2 คนนี้คุมเอง:

1. **order-svc (บุญ) ↔ product-svc (เดียร์)** — ตอน checkout ต้องเรียกเช็ค/จองสต็อกข้าม service กันโดยตรง
2. **order-svc / payment-svc (บุญ) → notification-svc (เดียร์)** — publish event `order.status_changed`, `payment.success` ผ่าน QStash ไปให้เดียร์รับ
3. **product-svc (เดียร์) → notification-svc (เดียร์เอง)** — `stock.updated` ไม่ข้ามคน แต่ยังต้องผูก payload ให้ตรงกับที่ order-svc ของบุญคาดหวัง (เช่น format event เดียวกัน)

ตกลง payload format ของ event (`stock.updated`, `order.status_changed`, `payment.success`) ให้ตรงกันตั้งแต่ต้น — เก็บไว้ใน `packages/types` เป็น Zod schema กลาง ทั้งบุญและเดียร์ import ใช้ร่วมกัน จะได้ไม่มีปัญหา payload ไม่ตรงกันตอน integrate

```
import { z } from "zod";

export const stockUpdatedEvent = z.object({
  event: z.literal("stock.updated"),
  productId: z.string().uuid(),
  beforeQty: z.number().int(),
  afterQty: z.number().int(),
});

export const orderStatusChangedEvent = z.object({
  event: z.literal("order.status_changed"),
  orderId: z.string().uuid(),
  customerId: z.string().uuid(),
  status: z.enum(["pending", "confirmed", "packed", "shipped", "delivered", "cancelled", "refunded"]),
});

export const paymentSuccessEvent = z.object({
  event: z.literal("payment.success"),
  orderId: z.string().uuid(),
  customerId: z.string().uuid(),
});

// subscriber ทั้งสองตัวที่ต้องฟัง event ชุดนี้ (QStash Topic fan-out)
export const QSTASH_SUBSCRIBERS = {
  notificationSvc: "/webhooks/qstash", // services/notification-svc
  reportSvc: "/webhooks/qstash",       // services/report-svc — เขตเขียนทีหลังได้ ไม่ block ใคร
};
```

## ลำดับความสำคัญ — ทำอะไรก่อนหลัง

### Phase 0 — เขตทำคนเดียวก่อน (ทุกคนรออันนี้)
1. Init Turborepo monorepo + `packages/config`, `packages/types`, `packages/database`, `packages/ui` (โครงเปล่า) เสร๊จแล้ว
2. Setup GitHub repo, invite บุญ/เดียร์, ตั้ง branch protection บน `main` เสร็จแล้ว
3. Setup Neon Postgres project + Cloudflare account + Wrangler CLI เสร็จแล้ว
4. Setup Kinde project (auth) เสร็จแล้ว เหลือเก็บ config ในโค้ดอีกนิดหน่อย
5. Push `docs/skill.md`, `design.md`, `structure.md`, `designsystem.md` เข้า repo ให้ทุกคนอ่านก่อนเริ่ม เสร็จแล้ว

> ระหว่าง Phase 0 บุญ/เดียร์ทำ Figma UI คู่ขนานไปได้เลย ไม่ต้องรอโค้ด บุญแก้เป็น light mode

### Phase 0.5 — ติดตั้ง mcp figma, memory

### Phase 1 — Foundational (ขนาน 3 คน)
- **เขต**: `auth-svc` (เชื่อม Kinde, JWT verify middleware) → `user-svc` (users, customers, staff, admins, addresses) ทำverify middle แล้วใส่ให้ทุกService แล้ว แต่ยังไม่ได้เทส ยังไม่ได้แต่ง kindeมันบอกว่าาเดี๋ยวจะมีเร็วๆนี้แหละถ้ากด urlไป https://musicgear.kinde.com musicgear

Welcome to musicgear
There's nothing to see here yet. But an awesome home page is coming soon.

Manage Kinde account จะขึ้นมา แสดงว่า Kinde project ยังไม่ถูกตั้งค่าให้ redirect กลับมาที่ localhost:3000 (หรือ URL ที่เราจะใช้ตอน dev) ให้เขตเข้าไปตั้งค่าใน Kinde dashboard ก่อนถึงจะ test login ได้จริงจัง 

ขอ 1 วันสำหรับเขตทำ auth-svc + user-svc ให้เสร็จ (login/logout, register, profile management, address management) — เพราะ service อื่นๆ ต้องเรียก user-svc เพื่อเช็คข้อมูล user/ลูกค้า/พนักงานก่อนถึงจะ test ได้เลย

- **เดียร์**: `product-svc`, `inventory-svc` แกนหลัก (products, categories, brands, productImages, inventory แบบพื้นฐานยังไม่มี log)
- **บุญ**: `cart-svc` (schema + Redis setup) — เริ่ม scaffold ได้เลย แต่เชื่อมจริงต้องรอ `product-svc` มี productId ให้อ้างอิงก่อน

⚠️ **`auth-svc` ต้องเสร็จก่อนใครทั้งหมด** เพราะทุก portal ต้อง login ผ่านก่อนถึงจะ test endpoint อื่นได้เลย — เขตควรดันตัวนี้เป็นอันดับ 1 จริงจัง

### Phase 2 — Integration เริ่มเข้มข้น
- **เดียร์**: เพิ่ม `inventoryLogs` + bundle endpoints บน `product-svc`
- **บุญ**: `order-svc` (เชื่อม `cart-svc` + เรียก `product-svc` เช็ค/จองสต็อก + ใช้ `addressId` จาก `user-svc`)
- **เขต**: เริ่มต่อ Admin Portal กับ `user-svc`/`product-svc` ที่มีแล้ว + เตรียม CI/CD auto-deploy

### Phase 3 — ปิด flow การซื้อ + แจ้งเตือน
- **บุญ**: `payment-svc` (เชื่อม Omise sandbox + `order-svc`)
- **เดียร์**: `notification-svc` (รับ event `stock.updated`, `order.status_changed`, `payment.success` ผ่าน QStash) — รอ `order-svc`/`payment-svc` ของบุญเสร็จก่อนถึง test event จริงได้ ระหว่างรอใช้ mock payload ตาม schema ใน `packages/types` ไปก่อน


- **เขต**: Staff/Customer portal ต่อ service ที่ทยอยเสร็จ, ดู deploy pipeline ให้นิ่ง

### Phase 4 — Report + ปิดงาน
- **เขต**: `report-svc` (ดึงข้อมูลจาก order/product/payment ที่เสร็จหมดแล้ว) — ตั้งใจให้ทำท้ายสุดเพราะต้องรอข้อมูลจริงจากทุก service ก่อนถึงจะ aggregate ได้ความหมาย
- **ทุกคน**: เก็บ frontend ให้ครบทุกหน้า, รัน UAT ตามตารางที่ทำไว้ก่อนหน้า (`musicgear_schema_and_uat.md`)

## ตาราง Service ↔ Frontend (เผื่อสับสนว่าใครต้องคุยกับ service ไหนตอนต่อ frontend)

| Portal | Service ที่ต้องเรียก |
|---|---|
| Customer Web (บุญ) | auth-svc, user-svc, product-svc, cart-svc, order-svc, payment-svc, notification-svc |
| Staff Portal (เดียร์) | auth-svc, product-svc, order-svc, notification-svc (trigger ฝั่ง backend เท่านั้น ไม่มี UI รับ) |
| Admin Portal (เขต) | auth-svc, user-svc, product-svc, report-svc |

ใครก็ตามที่ frontend ต้องเรียก service ของอีกคน ให้ดู API contract จาก `packages/api-client` + `packages/types` เป็นหลัก ไม่ต้องไปอ่านโค้ดข้างในของอีก service
